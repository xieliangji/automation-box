# Smart Monkey 参数调优手册（按目标落地）

本文给出当前主线（`main.py` + `SmartMonkeyAppRuntime`）的实用调参方法，重点回答：

- 不同测试目标下，参数该怎么配
- 调参时先动哪些，再动哪些
- 如何用 KPI 判断“调对了”

---

## 1. 先记住三个原则

1. 一次只改 1~2 个参数，避免“改了一堆但不知道谁生效”。
2. 用 baseline 对比：`run.benchmark_baseline_dir` 指向上一版 run 目录。
3. 分阶段放量：20 步冒烟 -> 100 步回归 -> 300+ 步验证。

---

## 2. 参数地图（知道每个旋钮影响什么）

## 2.1 Run 层

- `run.max_steps`：总预算；越大越容易收敛，但边际收益递减。
- `run.post_action_wait_ms`：动作节奏；越小越快，但抖动和误判风险会升高。
- `run.profile`：`functional`（稳健）/ `crash_stress`（高压）/ `monkey_compatible`（防脱防卡增强）。
- `run.crash_stress_*`：仅在 `crash_stress` 下控制 burst 与等待。

## 2.2 Policy 层

- `policy.epsilon`：探索比例（随机性）。
- `policy.top_k`：候选动作窗口宽度。
- `policy.no_progress_threshold`：卡住判定敏感度。
- `policy.enable_session_guardrails`：是否保护登录/表单会话。
- `policy.prefer_functional_pages`：是否偏向功能页。
- `policy.enable_login_bootstrap`：是否自动尝试登录。
- `policy.enable_pinch`：是否启用双指缩放动作（`pinch_in/pinch_out`）。
- `policy.pinch_zoom_context_boost`：在疑似可缩放场景中对 pinch 的加分幅度。
- `policy.pinch_non_zoom_penalty`：在非缩放场景中对 pinch 的惩罚幅度。
- `policy.monkey_*`：Monkey 兼容增强参数（防脱、防卡、风险冷却、随机抖动）。

## 2.3 Learning 层

- `learning.enabled`：是否启用 UCB 学习。
- `learning.alpha`：规则分与学习分的混合比例。
- `learning.ucb_exploration`：学习探索强度。
- `learning.persistence_enabled`：是否启用跨运行学习记忆。
- `learning.state_path`：学习状态持久化文件路径。
- `learning.min_observations_to_persist`：达到最小观测数后才写盘。
- `learning.module_bucket_enabled`：是否按模块聚类 arm。
- `learning.reward_* / penalty_*`：学习奖励与惩罚塑形。

## 2.4 Score Weights 层

- `score_weights.*`：规则评分九个因子的权重（`novelty/transition/depth/...`）。
- 建议只在前面三层调不动时再改，且单次改动幅度不超过 20%。

## 2.5 Sidecar 层（官方 monkey 双轨）

- `sidecar.monkey.enabled`：是否启用 Android 官方 monkey sidecar 轨道。
- `sidecar.monkey.step_interval`：每 N 个主轨 step 触发一批 sidecar。
- `sidecar.monkey.max_batches`：单次运行 sidecar 最大批次数。
- `sidecar.monkey.events_per_batch`：每批官方 monkey 注入事件数。
- `sidecar.monkey.throttle_ms`：官方 monkey 事件间隔。
- `sidecar.monkey.pct_touch/pct_motion/pct_nav/pct_syskeys`：官方 monkey 事件分布。

## 3. 四种目标的推荐配置

## 3.1 目标 A：功能覆盖优先（默认主场景）

适用：回归、日常巡检、尽量多跑业务页面。

推荐配置（在默认基础上）：

```yaml
run:
  profile: functional
  max_steps: 300   # 可在 300~500 间调
  post_action_wait_ms: 1000
policy:
  epsilon: 0.2
  top_k: 5
  prefer_functional_pages: true
  enable_session_guardrails: true
learning:
  enabled: false
```

关注 KPI：

- `unique_functional_states` 上升
- `functional_step_ratio` 高（通常 > 0.8）
- `out_of_app_ratio` 低（通常 < 0.05）

调参顺序：

1. 先调 `max_steps`（300 -> 500）
2. 再调 `post_action_wait_ms`（1200 -> 1000 -> 800）
3. 最后才动 `epsilon/top_k`

---

## 3.2 目标 B：Crash 发现优先（高压专项）

适用：稳定性专项、版本前高压巡检。

推荐配置：

```yaml
run:
  profile: crash_stress
  max_steps: 800          # 建议 600~1200
  crash_stress_wait_ms: 200
  crash_stress_burst_probability: 0.25   # 可在 0.2~0.4
  crash_stress_burst_min_steps: 2
  crash_stress_burst_max_steps: 5
policy:
  epsilon: 0.25
  top_k: 5
  enable_session_guardrails: true
learning:
  enabled: false
```

关注 KPI：

- `actions_per_minute` 明显提升
- `crash_per_1k_actions`、`time_to_first_crash_steps` 改善
- 同时看住 `out_of_app_ratio`（不要失控）

安全护栏建议：

- 先保持 `enable_session_guardrails=true`
- 若要更激进，先提高 burst 概率，再考虑降低等待
- 避免同时大幅降低等待 + 大幅提高 epsilon

---

## 3.3 目标 C：登录/账号链路专项

适用：专门测试登录页、账号页、会话切换问题。

推荐配置：

```yaml
run:
  profile: functional
  max_steps: 200
policy:
  prefer_functional_pages: false
  enable_session_guardrails: false
  enable_login_bootstrap: true
  bootstrap_max_attempts: 3
  bootstrap_retry_interval_steps: 20
learning:
  enabled: false
```

关注 KPI：

- `login_step_ratio`（应高于默认模式）
- 是否能稳定完成登录动作与后续跳转
- `out_of_app_ratio` 不应明显恶化

注意：该模式用于专项，不建议作为默认回归配置。

---

## 3.4 目标 D：学习收益优先（策略优化）

适用：要验证 UCB 学习是否真带来收益。

推荐配置：

```yaml
run:
  profile: functional
  max_steps: 300          # 建议先 300，再 500
policy:
  epsilon: 0.2
  top_k: 5
learning:
  enabled: true
  alpha: 0.8              # 可在 0.7~0.9
  ucb_exploration: 1.2    # 可在 0.8~1.6
  module_bucket_enabled: true
  reward_novel_state: 0.3
  penalty_recent_loop: 0.25
  penalty_system_action: 0.15
```

关注 KPI：

- `learning_exploration_rate` 下降（逐步收敛）
- `learning_average_reward` 上升并趋稳
- `learning_top_arms` 从分散变集中（模块化 arm）

实操经验（当前 `com.ugreen.iot`）：

- 学习收益临界点通常在 **约 300 steps**
- `300~500` 收益开始递减
- 若 `exploration_rate` 已低且 `average_reward` 稳定，再加步数性价比不高

---

## 3.5 目标 E：Monkey 兼容增强（防脱优先）

适用：希望引入部分官方 Monkey 风格随机机制，同时保留主线的风险控制与可观测性。

推荐配置：

```yaml
run:
  profile: monkey_compatible
  max_steps: 400
  monkey_wait_ms: 350
  monkey_ios_wait_ms: 260   # iOS 下优先使用
policy:
  monkey_out_of_app_streak_threshold: 2
  monkey_loop_streak_threshold: 3
  monkey_perturb_boost: 2.0
  monkey_risk_cooldown_steps: 3
  monkey_risk_penalty: 2.5
  monkey_score_jitter: 0.35
  monkey_diversity_state_repeat_threshold: 2
  monkey_diversity_state_repeat_penalty: 0.6
  monkey_diversity_novel_action_boost: 0.8
  monkey_diversity_frontier_boost: 0.6
  monkey_ios_permission_fastpath: true
  monkey_ios_permission_boost: 2.2
  monkey_ios_restart_penalty: 1.8
  monkey_ios_back_penalty: 0.5
  monkey_ios_swipe_boost: 0.6
  monkey_ios_list_swipe_boost: 0.8
  monkey_ios_pinch_boost: 0.5
  monkey_ios_static_text_click_penalty: 1.0
  monkey_ios_cell_click_boost: 0.7
  monkey_ios_back_like_click_penalty: 1.1
  monkey_ios_external_jump_penalty: 2.0
  monkey_ios_recovery_stuck_threshold: 10
  monkey_ios_same_state_recovery_threshold: 3
  monkey_ios_permission_recovery_grace_steps: 3
```

关注 KPI：

- `monkey_escape_boosted_ratio`（防卡/防脱触发强度）
- `monkey_risk_cooldown_ratio`（风险冷却是否生效）
- `monkey_max_out_of_app_streak`（越界是否被及时压制）
- `monkey_diversity_boosted_ratio`（多样性调分是否生效）
- `monkey_ios_permission_fastpath_ratio`（iOS 权限快路径命中）
- `permission_like_step_ratio`（权限弹窗停留）
- `monkey_ios_recovery_grace_step_ratio`（iOS 权限恢复缓冲命中）
- `out_of_app_ratio`、`recovery_success_rate`（整体稳定性）

---

## 3.6 目标 F：双轨执行（智能 + 官方 monkey）

适用：希望在主线可控探索上叠加官方 monkey 高频扰动，提升短时压力覆盖。

推荐配置：

```yaml
run:
  profile: functional
  max_steps: 300
sidecar:
  monkey:
    enabled: true
    step_interval: 15
    max_batches: 4
    events_per_batch: 35
    throttle_ms: 20
    pct_touch: 45
    pct_motion: 15
    pct_nav: 35
    pct_syskeys: 5
```

关注 KPI：

- `sidecar_batch_count`、`sidecar_success_rate`
- `sidecar_events_injected_total`
- `sidecar_recovery_count`（越低越好）
- 同时观察 `out_of_app_ratio` 与 `recovery_success_rate` 是否退化

## 4. 推荐调参流程（强执行）

### 步骤 1：建立基线

保持默认配置跑一轮，保存 `output/run_baseline_xxx`。

### 步骤 2：单目标单变量

每次只改 1~2 个参数，命名输出目录（如 `run_cov_tune_01`）。

### 步骤 3：看 KPI，不看主观感觉

优先看 `report/coverage_benchmark.json` 的对比字段：

- `comparison.*.delta`
- `composite_score`（综合参考，不替代专项指标）

### 步骤 4：通过门槛再放量

20 步通过后再 100 步，再 300+ 步。  
不要直接从 1000 步起跑。

---

## 5. 常见症状 -> 处理建议

### 症状 1：老在登录页打转

- 开 `prefer_functional_pages=true`
- 开 `enable_session_guardrails=true`
- 必要时开 `enable_login_bootstrap=true`

### 症状 2：探索太慢、动作不够激进

- `post_action_wait_ms` 逐步下调（每次 -100~-200）
- 或切 `run.profile=crash_stress`

### 症状 3：跳出 App 太频繁

- 提高 `penalty_out_of_app`（learning 模式）
- 提高 `no_progress_threshold` 或恢复默认
- 降低 `epsilon` 或 burst 概率

### 症状 4：学习不收敛

- 降低 `ucb_exploration`（如 1.2 -> 1.0）
- 提高 `alpha`（更信规则分）
- 确保 `module_bucket_enabled=true`（减少 arm 稀疏）

---

## 6. 推荐参数范围（便于快速起步）

- `run.max_steps`
  - 冒烟：20~50
  - 常规覆盖：200~500
  - crash 专项：600~1200
- `run.post_action_wait_ms`（functional）：800~1500
- `run.crash_stress_wait_ms`：120~300
- `run.monkey_wait_ms`：250~500
- `policy.epsilon`：0.15~0.30
- `policy.top_k`：4~7
- `learning.alpha`：0.7~0.9
- `learning.ucb_exploration`：0.8~1.6

---

## 7. 最后建议

对于团队协作，建议固定两套“标准预设”：

- `preset_functional_regression.yaml`
- `preset_crash_stress.yaml`

并规定 KPI 门槛（例如 `out_of_app_ratio`、`recovery_success_rate` 下限），这样每次版本迭代都能稳定比较，不会陷入“这次感觉还行”的主观评估。
