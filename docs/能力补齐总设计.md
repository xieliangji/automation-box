# Monkey 能力补齐总设计（工程化路线）

> 目标：把当前框架从“可用”推进到“同类领先”，补齐高频 crash 专项、学习型策略、多平台、与官方 monkey 联动能力。  
> 原则：默认行为稳定不变；所有新增能力开关化、可回滚、可观测。

---

## 1. 目标与成功标准

## 1.1 目标

- 提升问题发现效率（尤其 crash）
- 提升跨 App 泛化能力
- 提升运行稳定性与可复现性
- 补齐 iOS（WDA URL）路径
- 建立双轨执行（智能 + 官方 monkey）

## 1.2 成功标准（最终）

- 默认模式回归结果与当前基线一致
- crash 专项模式下 APM、crash 命中率、首次发现步数显著优化
- 学习策略在 A/B 上持续优于纯规则模式
- iOS 在不引入 Appium 前提下完成稳定闭环

---

## 2. 现状能力与缺口

## 2.1 已有能力（强项）

- 状态建模、动作提取/打分、UTG、恢复、报告闭环
- 会话护栏、登录引导、恢复校验、覆盖率 KPI
- 可解释日志与产物体系

## 2.2 核心缺口（更新）

- 动作频率偏保守，crash 压力不足
- 评分仍以静态权重为主，缺少在线学习
- 探索策略短视，长期收益不足
- 缺少与官方 monkey 的组合打法
- iOS 运行链路已完成，但短跑覆盖波动仍需通过参数与预算协同收敛

---

## 3. 设计原则

- **增量优先**：不改默认路径语义
- **开关化**：每个新能力均可启停
- **可观测**：新增能力必须有日志与指标
- **可回滚**：任一能力可一键回退
- **平台解耦**：驱动层封装，策略层复用

---

## 4. 总体架构增量

在现有链路上新增 4 个“增量能力面”：

1. 运行模式系统（profile）
2. 学习型策略插件（Bandit scorer augment）
3. 压测策略层（crash_stress action scheduler）
4. 双轨执行编排（official monkey sidecar）

并行推进 iOS：

5. iOS WDA 驱动链路（基于 `wda_url`）

---

## 5. 运行模式系统（Profile）

建议引入配置：

```yaml
run:
  profile: functional   # functional | crash_stress | coverage
```

每个 profile 管理一组参数模板：

- 动作节流
- 候选动作权重修正
- 恢复触发策略
- 风险放宽预算

默认 `functional` 与当前行为保持一致。

---

## 6. Crash 专项补齐（P1）

## 6.1 目标

提升高频交互引发 crash 的发现概率，同时保持可控。

## 6.2 核心策略

- 降低 `post_action_wait_ms`（设最小安全下限）
- 增加 burst 动作模板（连击/快速滑动/输入扰动）
- 恢复节流：避免过早恢复打断异常积累
- 护栏预算：只对非破坏性高风险动作做受控放宽

## 6.3 新增指标（报告）

- APM（actions per minute）
- burst 占比
- crash/1k actions
- time-to-first-crash
- issue replay 可复现率

## 6.1 Monkey 兼容增强（本轮落地，Android + iOS）

已落地到主线 `run.profile=monkey_compatible`，主要增强：

- 防脱：out_of_app 连续触发时加强逃逸动作评分
- 防卡：same_state 连续触发时加强扰动动作评分

## 7. 学习型策略补齐（P2）

## 7.1 混合评分

```
final_score = rule_score * α + learned_score * (1-α)
```

`α` 初期建议较高（如 0.8），随验证逐步放开。

## 7.2 候选算法

- UCB（首选，稳定）
- Thompson（后续）

## 7.3 学习输入

- 状态上下文：页面 flags、元素密度、历史访问频次
- 动作特征：类型、目标元素语义、风险标签
- 奖励定义：新状态发现、功能页覆盖、issue 信号、out_of_app 惩罚

## 7.4 上线策略

- 离线回放评估 -> 小流量 A/B -> 扩大比例

## 8. 探索策略补齐（P2）

## 8.1 UTG 前沿优先

## 8.2 分层策略

- 导航层：跨模块入口优先

## 9. 双轨执行补齐（P1-P2）

## 9.1 设计

新增“sidecar runner”思路：

- 轨道 A：智能 monkey（主）
- 轨道 B：官方 monkey（高频压力）

两轨都写入统一 run 汇总（可分目录），最后统一报告聚合。

## 9.2 官方 monkey 稳定策略

- 包约束 + 高风险事件比例收敛
- 分批执行 + 前台守护 + 自动拉回

## 13. 风险与约束

- 学习策略可能引入行为漂移
  - 对策：`α` 保守 + 灰度 + 可回滚
- crash_stress 可能导致 out_of_app 上升
  - 对策：预算控制 + 越界守卫
- 双轨执行增加复杂度
  - 对策：先做独立目录与最小聚合
