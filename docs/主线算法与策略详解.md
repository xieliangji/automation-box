# Smart Monkey 当前算法与策略详解（Android 主线）

本文面向开发和测试同学，说明当前主线（`main.py` + `SmartMonkeyAppRuntime`）中实际使用的算法与策略。

目标是回答三个问题：

- 现在系统到底“怎么决策”
- 每个策略影响哪类行为
- 该看哪些指标判断策略有效

---

## 1. 主线运行框架（先看总流程）

每一步（step）的核心顺序如下：

1. 采集状态：读取前台包名 / activity / hierarchy
2. 解析状态：提取 `UIElement` 与页面语义 flags
3. 生成候选动作：元素动作 + 系统动作（受护栏约束）
4. 规则打分：`ActionScorer` + `RuntimeActionScorer`
5. 学习打分（可选）：UCB 混合分
6. profile 调优（可选）：`crash_stress` 动作提权
7. 动作选择：`top_k + epsilon + 按分值加权随机`
8. 执行动作并采集 next_state
9. 计算 transition / reward / 统计 / 落盘
10. watchdog 检查 + 恢复策略（必要时触发）

这个顺序很关键：**学习和 crash_stress 都是在规则打分之后叠加，而不是替换原评分体系**。

## 2. 状态建模算法

### 2.1 层级树解析（HierarchyParser）

每个节点解析为 `UIElement`，保留：

- 元素属性：`class/resource_id/text/content_desc`
- 交互属性：`clickable/long_clickable/editable/scrollable`
- 空间属性：`bounds/depth/xpath/parent-child`

同时提取页面语义信号：

- `app_flags`：`login_page`、`list_page`、`form_page`、`webview`、`loading`
- `popup_flags`：如 `center_dialog`、`permission_like`
- `system_flags`：如 `permission_controller`、`settings`

这些 flags 是后续动作提取、打分、恢复、报告的重要输入。

### 2.2 状态指纹（StateFingerprinter）

维护两套 hash：

- `raw_hash`：保真调试用
- `stable_hash`：去抖+去重用

`stable_hash` 的关键归一化：

- 文本中的数字 -> `<NUM>`
- 时间 -> `<TIME>`
- 坐标按屏幕比例做桶化（0~10 桶）
- 元素签名排序后再哈希

最终 `state_id = stable_hash[:16]`，用于 UTG、恢复与报告索引。

## 3. 候选动作生成策略

### 3.1 动作来源

从可交互元素生成：

- `click`
- `long_click`（开关控制）
- `input`（开关控制，含输入内容启发式）
- `swipe`（对 scrollable）
- `pinch_in` / `pinch_out`（开关控制，双指缩放）

并补系统动作：

- `back`
- `wait`
- `restart_app`（开关控制）

### 3.2 高风险过滤 + 列表抽样

`RuntimeActionExtractor` 在基础提取上增加两类关键策略：

- 高风险过滤：黑名单关键词 + 内置高风险模式（如 logout/cancel_account/back_to_login）
- 列表抽样：对重复模板列表项做去重与代表采样（首/中/尾），避免动作爆炸

这部分本质是“控制动作空间”，让后续评分不被低价值重复项淹没。

### 3.3 会话护栏（Session Guardrails）

护栏策略已收敛在提取阶段：

- 在 `login_page/form_page` 上可阻断 `back`
- 登录态可疑页面可阻断 `restart_app`

目的：减少“回登录循环”或会话被破坏导致的探索失真。

## 4. 规则评分与动作选择

## 4.1 基础评分（ActionScorer）

总分结构：

`score = +novelty +transition +depth +business +escape +input -repeat -risk -stuck`

再乘以可配置权重（`score_weights.*`）。

关键项含义：

- `novelty`：动作历史执行越少越高
- `transition`：动作带来新状态的能力估计
- `depth`：是否可能进入更深层页面
- `business`：是否命中业务白名单/关键语义
- `escape`：卡住时对逃逸动作加分
- `repeat/risk/stuck`：重复、风险、卡死惩罚

### 4.2 运行期增强评分（RuntimeActionScorer）

在基础分上再做动态修正：

- 转移收益衰减（避免旧“高分动作”长期霸榜）
- 短环路惩罚（近期状态/动作循环）
- 历史卡死惩罚增强
- 列表页点击深度 bonus
- 功能页优先 bonus（`prefer_functional_pages=true`）
- 登录页非认证动作 penalty
- 缩放场景感知：在 map/image/webview/preview 等上下文提升 `pinch_in/pinch_out`，在非缩放上下文下调

### 4.3 动作选择器（不是纯贪心）

选择逻辑：

- 仅看前 `top_k`
- 以 `epsilon` 概率随机选（探索）
- 否则按分值做加权随机（利用）

这使策略既能收敛，也不容易过早陷入局部最优。

## 5. 学习策略（UCB Bandit）

> 默认关闭：`learning.enabled=false`。开启后是“规则分 + 学习分”混合，不会直接替换规则系统。

### 5.1 混合公式

`final_score = alpha * rule_score + (1 - alpha) * learned_ucb`

- `alpha` 越大，越偏规则
- `alpha` 越小，越偏学习

### 5.2 UCB 核心

每个 arm 维护：

- `count`（被选择次数）
- `reward_sum`（累计回报）
- `avg_reward`

UCB 打分：

`avg_reward + c * sqrt(log(total_observations) / count)`

其中 `c = learning.ucb_exploration`。

冷启动时未知 arm 返回固定先验分（当前实现为 `1.0`），保证新 arm 有探索机会。

### 5.3 arm 如何定义（经验复用的核心）

当前 arm key 是聚合语义桶，不是具体状态 ID：

`page_bucket | action_bucket | target_bucket | tag_bucket`

示例：

- `module_device|click|element|generic`
- `login|input|input|auth`

`page_bucket` 优先级：

1. `login_page` -> `login`
2. 若开启模块聚类：按 activity + 语义 token 映射到 `module_device/settings/account/message/scene/home`
3. 回退到 `form/list/webview/generic`

这就是“学习经验可复用”的根因：**同模块同类型动作共享统计，不会绑定到单一页面哈希。**

### 5.4 reward 重塑（Reward Shaping）

每步 reward 由多信号组合：

- 正奖励：状态变化、功能页、问题信号、新状态
- 负奖励：out_of_app、未变化、系统动作、近期原地循环

因此学习目标不是“盲目点更多”，而是“更可能带来有价值状态迁移”。

### 5.5 当前边界

- 经验复用范围：**单次 run 内**
- run 结束不做跨 run 持久化（后续可扩展）

## 6. Crash 专项策略（Profile）

当前有两个 profile：

- `functional`（默认）
- `crash_stress`

`crash_stress` 核心策略：

- 缩短动作后等待（`crash_stress_wait_ms`）
- burst 机制：随机窗口内提升 `click/swipe/input` 分数
- 对 `wait/back/restart_app` 做抑制
- 每步记录 `crash_stress_burst_active` 等审计字段

这让系统从“稳健探索”切换到“高扰动压测”。

---

## 7. 逃逸与恢复策略

### 7.1 触发条件

会触发恢复的主要场景：

- 已跳出目标包（`out_of_app=true`）
- 卡死/无进展（`stuck_score` 与阈值策略命中）

### 7.2 决策树（BacktrackHelper）

1. `out_of_app` -> `restart_app`
2. 轻度卡住 -> `press_back`
3. 重度卡住 -> `restart_to_checkpoint`
4. 无可用 checkpoint -> `restart_app`

支持 `avoid_login_checkpoint`，在功能优先模式下先避开登录检查点。

### 7.3 导航恢复（NavigationRecovery）

- 从 checkpoint 出发，结合 UTG 候选路径与 replay 历史动作
- 过滤“回登录倾向”动作（logout/back_to_login 等）
- 回放有限步导航动作
- 通过 `RecoveryValidator` 校验：
  - 是否命中 anchor/candidate
  - 是否回到目标包

---

## 8. Watchdog 与异常信号

watchdog 同时观察：

- logcat crash 信号（Fatal Exception + 目标包关联）
- logcat ANR 信号
- out_of_app
- 权限弹窗 / 系统设置页

并做事件去重后落盘到 `issues/*`，保留快照与上下文。

---

## 9. 可观测指标与产物

核心产物：

- `steps.jsonl`：每步动作、分数、学习字段、profile 字段
- `transitions.jsonl`：状态迁移
- `utg.json`：状态图
- `issues/*/summary.json`：异常详情
- `report/coverage_benchmark.json`：KPI 汇总与对比

关键 KPI：

- 覆盖类：`unique_states`、`unique_functional_states`
- 稳定类：`out_of_app_ratio`、`recovery_success_rate`
- crash 类：`actions_per_minute`、`crash_per_1k_actions`、`time_to_first_crash_steps`
- learning 类：`learning_exploration_rate`、`learning_average_reward`、`learning_top_arms`

---

## 10. 关键配置对照（建议重点掌握）

- 行为规模：`run.max_steps`
- 基础探索：`policy.epsilon`、`policy.top_k`
- 功能导向：`policy.prefer_functional_pages`
- 会话保护：`policy.enable_session_guardrails`
- 登录引导：`policy.enable_login_bootstrap` 与账号参数
- crash 专项：`run.profile=crash_stress` + burst 参数
- learning：`learning.enabled/alpha/ucb_exploration/module_bucket_enabled`
- reward shaping：`learning.reward_*`、`learning.penalty_*`

---

## 11. 实操建议（基于当前实现）

- 冒烟：20~50 steps（看链路是否通）
- 常规覆盖：200~500 steps
- crash_stress：600+ steps 更有意义
- learning 观察是否收敛：看 `exploration_rate` 下降、`average_reward` 稳定

如果 learning 指标已稳定，再继续加 steps 的边际收益通常会下降。

---

## 12. 一句话总结

当前 Smart Monkey 不是“随机点点点”，而是：

**状态语义建模 + 风险约束动作生成 + 多因子规则评分 + UCB 在线学习 + 分级恢复 + 全链路可观测** 的工程化探索系统。

它兼顾了可解释性、可控性和可演进性，适合作为长期主线持续迭代。
